# Claim Packet Stale After First-Slice docs/specs Edits — Debug
Date: 2026-06-20

## Problem

`npm run verify` fails at `lint · claim evidence state` (`claims:evidence-state:check`) with
`.cautilus/claims/status-summary.json is stale; run npm run claims:evidence-state`, even though this
session's slices touched no claim-source files.

## Correct Behavior

- Given the claim packet was discovered at some commit C and claim sources changed in C..HEAD,
- when `claims:evidence-state:check` runs,
- then it must report the packet stale until the packet is refreshed against HEAD — and a slice that
  commits claim-source changes must refresh the packet so the committed `status-summary.json` honestly
  reflects HEAD and the gate passes.

## Observed Facts

- Verify stopped at `claims:evidence-state:check` (exit 1); all earlier phases (incl. `lint:specs`,
  `specdown:project:check`, `specdown:claim-state:check`) were green.
- This session's commits/working tree changed only `charness-artifacts/**`, `scripts/check-specs.mjs`,
  and `scripts/check-specs.test.mjs` — no claim sources.
- Committed `status-summary.json` recorded `packetGitCommit = currentGitCommit = 80e051a`,
  `isStale=false` — but HEAD is `b082597a` (`80e051a` is an ancestor).
- `render-claim-evidence-state.mjs --check` normalizes by deleting `currentGitCommit`, `headDrift`,
  `changedSources`, `comparisonStatus`, then fails if `isStale===true` or `changedSourceCount!==0` on
  either the committed or the freshly-recomputed git-state.

## Reproduction

1. `node scripts/agent-runtime/render-claim-evidence-state.mjs --refresh-status --check` → stale.
2. Recompute via `npm run claims:evidence-state`, then inspect `gitState`:
   `changedSources = ["docs/specs/evidence/index.spec.md", "docs/specs/ledger/index.spec.md"]`,
   `changedSourceCount=2`, `isStale=true`, `headDrift=true`, `packetGitCommit=80e051a`,
   `currentGitCommit=b082597a`.

## Candidate Causes

- (control-flow) A claim source changed in `packetGitCommit..HEAD` that this session did not author.
- (state) Pure head-drift (HEAD moved, no claim source changed) — tolerated by the check; would NOT fail.
- (env/tooling) A genuine bug in the staleness comparison (false positive on head-drift).

## Hypothesis

The first slice (`6dd78e77`) committed edits to `docs/specs/evidence/index.spec.md` and
`docs/specs/ledger/index.spec.md` (the projected-claim-state links) — both are claim sources — WITHOUT
running `claims:refresh:all`, so the packet (pinned at `80e051a`) went stale. The prior session believed
the first slice touched no claim sources ("미편집 → 불필요했음"), which was wrong. If true,
`claims:evidence-state` will report exactly those two files as `changedSources`, and `claims:refresh:all`
will clear it; if it were pure head-drift the recomputed `changedSourceCount` would be 0 and the check
would already pass.

## Verification

- Disconfirmer (cheapest falsifier): `claims:evidence-state` recompute reported
  `changedSources=[evidence/index.spec.md, ledger/index.spec.md]`, `changedSourceCount=2` — NOT 0 — so it
  is a real claim-source change, not head-drift. Hypothesis confirmed, "pure head-drift" candidate refuted.
- `npm run claims:refresh:all` (prescribed when claim sources change) → `isStale=false`,
  `comparisonStatus=fresh`, `packetGitCommit=currentGitCommit=b082597a`, `changedSourceCount=0`.
- `candidateCount` unchanged at 398 before/after — a clean re-anchor, no semantic claim change (SC6 safe;
  no badge input moved).
- `claims:evidence-state:check` and `claims:status-report:check` both PASS after refresh.

## Root Cause

A claim-source edit (the two `docs/specs/*index.spec.md` link additions) landed in the committed first
slice without the required `claims:refresh:all`, because the prior session misclassified `docs/specs/**`
pages as non-claim-sources. The committed `status-summary.json` therefore carried git-state pinned to a
pre-edit commit (`80e051a`), and `--refresh-status --check` surfaced the drift on the first post-commit
verify (this session). The fix is the prescribed packet refresh against HEAD.

## Invariant Proof

- Invariant: the committed claim packet's `gitState` must report `changedSourceCount=0`/`isStale=false`
  against HEAD (packet freshness = no un-refreshed claim-source diff between `packetGitCommit` and HEAD).
- Producer Proof: `discover claims` re-discovered against HEAD; `latest.json` packetGitCommit=b082597a.
- Final-Consumer Proof: `claims:evidence-state:check` + `claims:status-report:check` pass; full
  `npm run verify` green (recorded at Phase 3 closeout).
- Interface-Shape Sibling Scan: `evidenced-typed-runners.json`, `canonical-claim-map.json`,
  `evidence-state.json`, `claim-status-report.md`, `claim-evidence-state.md` all regenerated by the same
  `claims:refresh:all` chain, so no downstream view keeps the stale basis.
- Non-Claims: did not change `workingTreePolicy: excluded`; did not alter any claim verdict or badge.

## Detection Gap

- Surface: `npm run verify` (`claims:evidence-state:check`). It DOES catch this — but only on a verify run
  AFTER the claim-source change is committed. The git-state staleness uses committed-diff
  (`workingTreePolicy: excluded`), so the slice that introduces the change passes its own pre-commit verify
  (working tree excluded), commits, and ships latent staleness that the NEXT verify/push catches.
- What did not fire: the first slice's pre-commit verify (working-tree changes excluded) and the fact that
  verify was not re-run post-commit; the standing pre-push claim-freshness gate would have caught it at
  push, but the work is unpushed.
- Smallest change to fire it at authoring time: `follow-up: claim-freshness-precommit` — a pre-commit hook
  (or a verify mode) that evaluates claim-source freshness including staged/working-tree changes, so a
  claim-source edit forces the refresh in the same slice rather than the next one.

## Sibling Search

- Mental model (wrong): "`docs/specs/**` pages are spec views, not claim-discovery sources, so editing them
  does not require `claims:refresh:all`." Reality: the discovery boundary is README/AGENTS/CLAUDE + linked
  docs, which reaches `docs/specs/**`, so they ARE claim sources.
- commit-history axis: other unpushed commits in `80e051a..HEAD` that touched claim sources
  (`docs/specs/audit/*.json`, `docs/specs/evidence/claim-evidence-state.md`) | decision: absorbed by the
  single `claims:refresh:all` re-discovery against HEAD | proof: `changedSourceCount=0` post-refresh.
- doc-classification axis: any future slice editing `docs/specs/**` | decision: must run
  `claims:refresh:all` before commit/push | proof: this incident.
- cross-file: `follow-up: claim-freshness-precommit` (Detection Gap) is the durable sibling — a gate that
  catches claim-source staleness in the authoring slice, not the next one.
