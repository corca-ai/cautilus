# HITL Runtime Sync Metadata Debug
Date: 2026-05-08

## Problem

After `charness` was updated, resuming the active HITL session failed before the next Claim Discovery chunk could be presented.

## Correct Behavior

Given an existing HITL runtime session is still the active review state, when the updated `hitl` gate runs, then the session should name accepted rules, copy the current chunk's active rules into `active_rules_applied`, record target/cursor/line-bound evidence, and sync `charness-artifacts/hitl/latest.md` with `hitl-runtime-sync` metadata before closeout or handoff.

## Observed Facts

- `check_review_state.py --phase pre-edit` returned `accepted rules are present but active_rules_applied is empty`, `target_cursor_checked must be true before editing or advancing`, and `target_cursor_check_result must name target, chunk, queue item, line bounds, and epoch`.
- `sync_review_artifact.py --check` returned `durable artifact is missing hitl-runtime-sync metadata`.
- The active session was `.charness/hitl/runtime/hitl-20260508-062748`.
- `state.yaml` pointed at `docs/specs/user/claim-discovery.spec.md` and `claim-discover-heuristics-and-dedupe-evidence`, but it did not yet include the new approval-boundary fields required by Charness 0.5.19.
- `rules.yaml` held only `show-rewritten-chunk-after-edit`; the broader accepted rules were still recoverable from `hitl-scratchpad.md` and the previous durable HITL artifact.

## Reproduction

Run:

```bash
python3 /home/hwidong/.codex/plugins/cache/local/charness/0.5.19/skills/hitl/scripts/check_review_state.py --repo-root . --session-id hitl-20260508-062748 --phase pre-edit
python3 /home/hwidong/.codex/plugins/cache/local/charness/0.5.19/skills/hitl/scripts/sync_review_artifact.py --repo-root . --session-id hitl-20260508-062748 --check
```

## Candidate Causes

- The runtime session was created before the updated HITL contract required explicit active approval-boundary fields.
- The durable artifact was manually maintained in the older summary shape and had not yet been regenerated by `sync_review_artifact.py`.
- Accepted rules were split across `rules.yaml`, `hitl-scratchpad.md`, and the old durable artifact, so the new gate could not infer the active subset safely.

## Hypothesis

If the current session state records accepted rules, copies the current chunk's relevant rules into `active_rules_applied`, records a cursor check naming target, chunk, queue item, lines, and epoch, and then regenerates the durable HITL artifact with `sync_review_artifact.py`, both the runtime gate and freshness check should pass without changing the review target text.

## Verification

- Added accepted rules, `active_rules_applied`, `target_cursor_checked: true`, and a target/cursor check result for `docs/specs/user/claim-discovery.spec.md` lines 62-129 to `.charness/hitl/runtime/hitl-20260508-062748/state.yaml`.
- `check_review_state.py --phase pre-edit` passed.
- `check_review_state.py --phase cursor-advance` passed.
- `sync_review_artifact.py --session-id hitl-20260508-062748` regenerated `charness-artifacts/hitl/latest.md`.
- `sync_review_artifact.py --check` returned `status: current`.

## Root Cause

The handoff was accurate about the active review target, but the runtime state still used the pre-update HITL state shape.
Charness 0.5.19 added stricter gates and durable sync metadata, so the existing session needed a state migration before the next chunk could be reviewed.

## Seam Risk

- Interrupt ID: hitl-runtime-sync-metadata
- Risk Class: contract-freeze-risk
- Seam: Charness public HITL skill state contract across installed-version updates
- Disproving Observation: A freshly bootstrapped Charness 0.5.19 HITL session would already seed the new fields and sync metadata.
- What Local Reasoning Cannot Prove: Whether every older HITL session in other repos has enough scratchpad state to reconstruct accepted rules without user confirmation.
- Generalization Pressure: monitor

## Interrupt Decision

- Premortem Required: no
- Next Step: impl
- Handoff Artifact: `charness-artifacts/hitl/latest.md`

## Prevention

When resuming a HITL session after a Charness update, run `check_review_state.py --phase pre-edit` and `sync_review_artifact.py --check` before presenting the next chunk.
If either fails on missing approval-boundary metadata, migrate the active session state from the scratchpad and durable artifact before continuing.
