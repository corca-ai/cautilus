## Symptom

(https://corca-ai.github.io/cautilus/user/doctor-readiness.html),

```run:shell
# Show the full sample adapter YAML generated by Cautilus.
```

Instead the rendered output shows only the variable-binding line, shaped like:

`$sample_cautilus=/tmp/tmp.fbicr9GgOP/cautilus, $sample_repo=/tmp/tmp.fbicr9GgOP`

## Hypothesis space (falsify before fixing)

1. specdown variable scoping: `${sample_repo}` is not propagated into this block's shell,
   so `cat` fails (empty/wrong path) and the runner surfaces the prior block's
   variable-binding line as this block's "output".
2. Render/adapter block-capture off-by-one: stdout is associated with the wrong block,
   so the variable-binding display leaks into the adapter-YAML block's output slot.
3. The variable-binding line is specdown's own display of captured vars, and it is
   being rendered IN PLACE OF the block body output rather than alongside it.

The `$name=value, $name=value` shape strongly suggests this is specdown's variable-binding
render, not a shell stdout — point investigation at the executable-spec runner first,
then the repo render adapter.

## Correction (2026-06-09, charness:debug)

Status updated: investigated; the "missing trace config" path is FALSIFIED. Still open, direction corrected.

Falsified: the specdown report message `"no trace configuration found in config"` is UNRELATED.
specdown's `trace` config drives the `specdown trace` traceability-graph subcommand (document/requirement edges),
which is non-fatal ("a trace error does not prevent spec execution") and has nothing to do with `run:shell -> $var` capture.
Adding a `trace` block to specdown.json would NOT fix the render.

Corrected direction (from the installed specdown skill docs — `specdown install skills` writes `.agents/skills/specdown/`):
- syntax.md "Summary Lines": when a `run:` block's first line is a comment (`#`, `//`, `--`),
  specdown takes it as the block's summary and renders the block COLLAPSED in HTML — only the summary + pass/fail show.
  The cat block leads with `# Show the full sample adapter YAML generated by Cautilus.`,
  so the adapter YAML is collapsed behind the fold; the visible `$name=value` line is specdown's variable-binding display.
- Most likely this is a SPEC-AUTHORING issue, not a specdown bug or a config change:
  drop/relocate the leading comment so the cat stdout renders inline (or use a `>` marker / non-comment first line).
- Confirm with a minimal local repro: `specdown run` over a tiny spec that captures `-> $x` then a comment-led block using `${x}`,
  and inspect the HTML report for that block (collapsed vs inline output).

Reference docs (read these, do not reverse-engineer the stripped binary): specdown skill `syntax.md`, `config.md`, `report.md`.
