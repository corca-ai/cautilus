# Debug Review: stale claim packet schema
Date: 2026-04-27

## Problem

`./bin/cautilus claim show --input /tmp/cautilus-claims-self.json --output /tmp/cautilus-claim-status-summary.json` failed with:

```text
claimCandidates[0].claimFingerprint must be a non-empty string
exit status 1
```

## Correct Behavior

Given a no-input `$cautilus` route finds an existing `cautilus.claim_proof_plan.v1` packet, when it asks the current CLI to summarize that packet, then either the packet should validate under the current schema or the route should classify it as stale and ask before running a fresh discovery scan.

## Observed Facts

- The current checkout launcher resolves Cautilus v0.13.0 from the source checkout.
- `./bin/cautilus healthcheck --json` reports a healthy command registry with 55 registered commands.
- `./bin/cautilus doctor --repo-root . --scope agent-surface` reports `ready=true`.
- `./bin/cautilus adapter resolve --repo-root .` finds `.agents/cautilus-adapter.yaml` and reports it valid.
- The repo has no checked-in non-fixture claim proof-plan packet.
- `/tmp/cautilus-claims-self.json` exists from 2026-04-26 and has `schemaVersion=cautilus.claim_proof_plan.v1`.
- That packet has `candidateCount=561`.
- All 561 candidates are missing `claimFingerprint`, `evidenceStatus`, `reviewStatus`, and `lifecycle`.
- `./bin/cautilus claim validate --claims /tmp/cautilus-claims-self.json --output /tmp/cautilus-old-claim-validation.json` emits `valid=false` with the same `claimFingerprint` issue.
- `./bin/cautilus claim show --input fixtures/claim-discovery/example-proof-plan.json --output /tmp/cautilus-example-claim-status-summary.json` succeeds, so the current command works for a current-shaped packet.
- A web search for the exact error string found no public matches, which is expected for a local product packet validation message.

## Reproduction

Run:

```bash
./bin/cautilus claim show --input /tmp/cautilus-claims-self.json --output /tmp/cautilus-claim-status-summary.json
```

The command exits non-zero with:

```text
claimCandidates[0].claimFingerprint must be a non-empty string
exit status 1
```

## Candidate Causes

- The `/tmp` packet was generated before the claim proof-plan schema gained required claim fingerprint and review state fields.
- The command is reading a corrupt or manually edited packet that happens to keep the same schema version.
- The current validator is stricter than the old discover output while the schema version did not change, so old same-version packets cannot be summarized.

## Hypothesis

If the `/tmp` packet is stale rather than corrupt, then a current-shaped checked-in fixture should pass `claim show`, while the old packet should fail validation on fields that are now required across candidates.

## Verification

The current fixture `fixtures/claim-discovery/example-proof-plan.json` passes `claim show` and writes `/tmp/cautilus-example-claim-status-summary.json`.
The old `/tmp/cautilus-claims-self.json` fails both `claim show` and `claim validate`.
`jq` confirms all 561 candidates in the old packet lack the current candidate state fields.

## Root Cause

The no-input route found a temp claim proof-plan packet generated by an older claim discovery slice.
The packet kept the same `cautilus.claim_proof_plan.v1` schema version but does not satisfy the current v0.13.0 validator requirements for per-claim fingerprints and state fields.

## Prevention

Treat old temp claim packets as stale when `claim show` rejects them.
For no-input routing, ask before running a fresh `claim discover` scan rather than silently trusting a stale `/tmp` packet.
If same-version packet compatibility becomes important, add a migration or clearer packet version bump instead of relying on temp artifact memory.
